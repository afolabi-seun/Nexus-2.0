# Authentication & Security

Login flow, JWT issuance, refresh token rotation, session management, OTP, account lockout, anomaly detection, and service-to-service authentication.

## Login Flow

`SecurityService.Infrastructure/Services/Auth/AuthService.LoginAsync()`:

```
1. Check lockout         → AccountLockedException (423)
2. Resolve user          → TeamMember via ProfileService, fallback to PlatformAdmin
3. Check FlgStatus       → AccountInactiveException if Suspended/Deactivated
4. BCrypt verify         → InvalidCredentialsException, increment lockout counter
5. Anomaly detection     → SuspiciousLoginException if new IP
6. Reset lockout         → Clear Redis lockout keys on success
7. Generate tokens       → JWT access token + opaque refresh token
8. Create session        → Redis session with device ID, IP, JTI, expiry
9. Store refresh token   → BCrypt-hashed in Redis (refresh:{userId}:{deviceId})
10. Publish audit event  → Outbox → UtilityService
```

Returns `AuthResult` with `AccessToken`, `RefreshToken`, `ExpiresIn`, `IsFirstTimeUser`.

## JWT Access Token

Generated by `JwtService.GenerateAccessToken()`. Claims include:

| Claim | Source | Example |
|-------|--------|---------|
| `sub` / `userId` | TeamMember ID | `a1b2c3d4-...` |
| `organizationId` | From ProfileService | `org-uuid` |
| `departmentId` | Primary department | `dept-uuid` |
| `roleName` | Role assignment | `OrgAdmin` |
| `departmentRole` | Department-level role | `DeptLead` |
| `email` | User email | `user@example.com` |
| `firstName`, `lastName` | User profile | `Jane`, `Doe` |
| `deviceId` | Login device | `device-123` |
| `jti` | Unique token ID | `guid` |
| `isFirstTimeUser` | First login flag | `true` |

Expiry: configurable via `AppSettings.AccessTokenExpiryMinutes`.

## Refresh Token Rotation

`AuthService.RefreshAsync()`:

1. Extract `userId` and `deviceId` from the expired access token
2. Look up `refresh:{userId}:{deviceId}` in Redis
3. BCrypt verify the provided refresh token against the stored hash
4. If mismatch → `RefreshTokenReuseException` (potential token theft — all sessions revoked)
5. Generate new access + refresh token pair
6. Replace the stored refresh hash in Redis
7. Return new `AuthResult`

The refresh token is an opaque random string (not a JWT). Only its BCrypt hash is stored.

## Token Blacklist

`TokenBlacklistMiddleware` runs after JWT authentication:

1. Extracts `jti` from the validated JWT
2. Checks Redis key `blacklist:{jti}`
3. If exists → `TokenRevokedException` (401)

Tokens are blacklisted on logout and password change. TTL matches the token's remaining lifetime.

## Session Management

`SessionService` manages sessions in Redis:

- Key: `session:{userId}:{deviceId}`
- Value: JSON with `jti`, `ipAddress`, `createdAt`, `expiresAt`
- TTL: matches access token expiry

Sessions are created on login, validated on each request via `JwtClaimsMiddleware`, and destroyed on logout.

## Account Lockout

Redis-based lockout with configurable thresholds:

```
lockout:{email}         → attempt counter (TTL: AccountLockoutWindowHours)
lockout:locked:{email}  → lock flag (TTL: AccountLockoutDurationMinutes)
```

Flow:
1. Failed login → increment `lockout:{email}`
2. First failure → set TTL on the counter
3. Counter reaches `AccountLockoutMaxAttempts` → set `lockout:locked:{email}`, delete counter
4. Subsequent login attempts → `AccountLockedException` (423)
5. Lock auto-expires after `AccountLockoutDurationMinutes`
6. Successful login → delete lockout counter, add IP to trusted list

## OTP (One-Time Password)

`OtpService` for two-factor verification:

- Stored in Redis: `otp:{userId}:{purpose}` with configurable TTL
- BCrypt-hashed before storage
- Max attempts tracked: `otp_attempts:{userId}:{purpose}`
- Exceeding max attempts → `OtpMaxAttemptsException`
- Expired OTP → `OtpExpiredException`
- Wrong code → `OtpVerificationFailedException`

## First-Time User Flow

`FirstTimeUserMiddleware` enforces password change on first login:

1. Checks `isFirstTimeUser` claim in JWT
2. If `true`, only allows access to password change endpoint
3. All other endpoints → `FirstTimeUserRestrictedException`
4. After password change, `isFirstTimeUser` is set to `false` via ProfileService

## Anomaly Detection

`AnomalyDetectionService` tracks login patterns:

- Maintains trusted IP list per user in Redis
- New IP on login → `SuspiciousLoginException` (or logs warning depending on config)
- Successful login from new IP → adds to trusted list

## Service-to-Service Authentication

Internal service calls use dedicated JWT tokens issued by `ServiceTokenService`:

1. Calling service requests a token via `ServiceTokenService.IssueTokenAsync(serviceId, serviceName)`
2. Token is a JWT with `serviceId` claim (no user claims)
3. BCrypt hash stored in `service_tokens` DB table
4. Raw token cached in Redis: `service_token:{serviceId}` (23-hour TTL)
5. Receiving service validates via `ServiceTokenService.ValidateServiceTokenAsync()`
6. ACL check: only `ProfileService`, `WorkService`, `UtilityService`, `SecurityService` are allowed

Service clients cache the token in memory with a 30-second buffer before expiry:

```csharp
if (_cachedToken is not null && DateTime.UtcNow.AddSeconds(30) < _tokenExpiry)
    // reuse cached token
```

`RoleAuthorizationMiddleware` skips role checks for service-auth tokens (identified by `serviceId` in `HttpContext.Items`).

## Middleware Pipeline Order

```
1.  CORS
2.  Serilog request logging
3.  CorrelationIdMiddleware
4.  GlobalExceptionHandlerMiddleware
5.  RateLimiterMiddleware
6.  Routing
7.  Authentication (JWT Bearer)
8.  Authorization
9.  JwtClaimsMiddleware
10. TokenBlacklistMiddleware
11. FirstTimeUserMiddleware (SecurityService, ProfileService only)
12. RoleAuthorizationMiddleware
13. OrganizationScopeMiddleware
14. Controllers
```
